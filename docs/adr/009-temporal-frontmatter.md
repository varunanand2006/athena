# ADR 009 — Temporal Memory via Note Frontmatter

**Date:** 2026-06-16
**Status:** Accepted
**Relates to:** Phase 17, ADR 007 (vault substrate), ADR 008 (auto capture), ADR 004 (summary-RAG)

---

## Context

The memory vault (Phases 14–16) stores durable facts as prose notes and now
loads the whole vault into the agent's context for recall. What it cannot do is
answer **time-based** questions — "what's coming up this week?", "any deadlines
soon?" — because dates are buried in free text with no queryable structure.

Phase 17 adds that capability. The design question is *where the date lives*: in
the note, or in a separate store the agent can query by date range.

---

## Decision

### 1. One store: the date lives in the note's frontmatter

A note that involves a date carries it in its own YAML frontmatter as an
optional `events` field — a list of `{date: YYYY-MM-DD, kind: <short>}` maps.
There is **no facts table and no parallel store**.

Why one store:

- **No dual-store drift.** A note plus a separate row that mirrors its date can
  drift on edit/delete — the row says one thing, the note another. Keeping the
  date *inside* the note means exactly one record: delete the note and the event
  is gone; edit the date and it's edited everywhere. We won't pay the sync tax.
- **The format is still the API.** `events` is just more frontmatter — the same
  files render in the frontend and in Obsidian, and the same Phase 14 parse
  helpers carry it. No schema, no migration. (Same reasoning as
  [ADR 007](007-agent-memory-vault.md) §1.)
- **Derived and rebuildable.** `events` is extracted *from* the note's prose, so
  it can be regenerated by re-scanning the vault. That makes the structured
  field a cache in front of the source-of-truth prose, never a second master.

### 2. We don't classify "facts" — we extract one attribute: a date

We explicitly do **not** split memories into "facts" vs "prose." Factness
depends on the *future query*, not the content — the same note is a "fact" to
one question and background to another — so the classification is unanswerable.

Instead, **every memory stays prose**, and we extract exactly **one** queryable
attribute when it's present: a date. Reflection's decision is the narrow,
mechanical *"does this note contain a date/deadline?"* — never the philosophical
*"is this a fact?"*. Only concrete, resolved calendar dates are captured;
"next Friday" is resolved against today's date, and vague/unresolvable timing is
left as prose only (never forced into a malformed event).

Getting it wrong is cheap and recoverable: a missed date just means the note
isn't time-queryable (it still exists as prose), and because `events` is derived
it can be rebuilt by re-scanning. Early extraction mistakes are never permanent.

### 3. Lineage: thin queryable index in front of rich content

This is the same pattern as summary-based RAG
([ADR 004](004-summary-based-rag.md)): a small structured/queryable key (there, a
summary vector; here, an `events` date) sits in front of the rich content (the
full document / the prose note), and the content remains the thing you actually
read and the source of truth. We are reusing a proven shape, not inventing one.

### 4. Calendar export is a future, downstream concern

Pushing `events` to Google Calendar (or any external calendar) is explicitly
**out of scope**. It would be a one-way downstream *export* built on top of the
`events` field — additive later, and not a reason to change the storage decision
now. The vault stays the source of truth; a calendar would be a projection of
it, never the other way around.

---

## Alternatives considered

| Option | Rejected because |
| ------ | ---------------- |
| A separate `events`/`facts` table (or Qdrant collection) keyed by date | Dual-store sync: the row can drift from the note on edit/delete; needs its own cleanup on note delete; reintroduces the exact two-master problem ADR 007 avoided. The note is already the record — put the date in it. |
| Classify memories as structured "facts" vs unstructured "prose" | Factness depends on the query, not the content — unanswerable. Forces a brittle taxonomy and a schema we'd fight to maintain. Extracting one attribute (a date) sidesteps the whole question. |
| Parse dates out of prose at query time (no stored field) | Re-parsing every note's free text on every time query is slow and fragile (NLP date extraction, relative-date resolution against the wrong "now"). Resolve once at capture, store the concrete date. |
| Build a derived date index now for fast range queries | Premature at a handful of notes; `upcoming()` is a linear full-vault scan with an explicit tripwire (`MEMORY_EVENTS_MAX_NOTES`) that names when an index becomes worth it. |
| Sync events to Google Calendar in this phase | Out of scope; a downstream export built on `events`, not a storage decision. Adds an external dependency and write-back/conflict questions we don't need yet. |

---

## Consequences

- **Time queries work with zero new infrastructure.** `upcoming(timeframe)`
  scans `events` frontmatter across the vault and returns dated hits, sorted —
  the same full-vault-scan shape as Phase 16's full-context load.
- **Capture stays conservative and origin-honest.** Reflection only emits
  resolved dates; `_sanitize_events` drops anything non-ISO so a bad date never
  lands as a broken event. The `source` field (ADR 008) is unaffected.
- **Re-reflection is safe.** `events` is merged across same-slug updates
  (dedup on `(date, kind)`), like tags — extending a conversation adds new
  deadlines without dropping or duplicating old ones.
- **The tripwire names the next phase.** When `upcoming()` trips
  `MEMORY_EVENTS_MAX_NOTES`, that's the honest signal to build a derived date
  index — not a guess made today.
- **Obsidian-compatible.** `events` is valid YAML flow frontmatter; the vault
  still opens cleanly as a vault.
