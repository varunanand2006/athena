# ADR 010 — Interlinked Memory (the wiki graph)

**Date:** 2026-06-16
**Status:** Accepted
**Relates to:** Phase 18; ADR 007 (vault), ADR 008 (auto capture), ADR 009 (events), ADR 004 (summary-RAG)

---

## Context

The memory vault (Phases 14–17) stores durable facts as prose notes, captures
them automatically (reflection), loads the whole vault for recall (Phase 16),
and extracts dates (Phase 17). What it lacks is **structure between notes**:
every note is an island. Inspired by Andrej Karpathy's "LLM Wiki" pattern
(raw sources → an agent-maintained, interlinked Markdown wiki the model reads
for pre-built context), Phase 18 turns the vault into a **graph**: notes
cross-link with `[[wikilinks]]`, concept/entity pages emerge, and synthesis
reconciles pages instead of only appending.

A mapping check first: Athena already had most of the Karpathy substrate — the
documents pipeline is the raw/inbox layer, the memory vault is the synthesized
wiki layer, CLAUDE.md + the reflection prompt is the `agents.md` schema, and
Phase 16's ambient load is "the agent reads the wiki for context." The missing
pieces were the **edges** (links/backlinks), **concept pages**, and
**reconciliation**. Phase 18 builds exactly those, scoped to the existing
conversation-reflection path ("graph only"); feeding the wiki from documents is
deferred to the next phase.

---

## Decisions

### 1. The graph is derived from prose, not stored in a second place

`[[wikilinks]]` live in note **bodies**; edges are **computed by scanning**
(`extract_links`, `backlinks`), never persisted in a link table or graph DB.

Why:

- **One source of truth.** Same reasoning as ADR 009's rejection of a separate
  events table: a link table mirroring the prose can drift on edit/delete. Keep
  the link in the body and the edge is always exactly what the text says.
- **Rebuildable.** The whole graph is a pure function of the note bodies, so it
  can be regenerated by re-scanning — early/rough links are never permanent
  state to migrate.
- **Obsidian-native.** `[[...]]` is Obsidian's own link syntax, so the same
  files give a real graph view in Obsidian with zero extra machinery.

Tradeoff: backlinks are an O(n) scan, not an indexed lookup. Trivial at this
vault size and consistent with the Phase 16/17 full-vault-scan pattern; revisit
with a derived index only at the same scale where embeddings become necessary.

### 2. Link identity is the slug

A `[[Target]]` link resolves to a note via `slugify(Target)` — the **same**
function that defines note identity (ADR 007 §2). So linking, creating, and
updating all key off one rule: `[[Meta interview prep]]`,
`meta-interview-prep.md`, and a future write titled "Meta interview prep" are
the same node. `[[Target|Display]]` (Obsidian alias syntax) is supported; the
target before `|` is what resolves.

### 3. Reconcile concept pages (replace); append personal-fact notes

Update semantics now depend on note kind:

- **Concept/entity pages** (`concept: true` in a reflection decision) are
  **reconciled** — synthesis returns the full current page and `write_note(...,
  replace=True)` rewrites the body. A wiki page should read as one coherent
  page, not an ever-growing append log.
- **Personal-fact notes** keep the **append** behavior from Phase 15 (dated
  additions), preserving their audit trail.

Destructive rewrite is made safe by an **operation log** (`_log.md`, Karpathy's
`log.md`): every synthesis write is recorded with a timestamp, so *what changed*
survives even when a page body is overwritten. Combined with the user's
per-note delete control (ADR 008 §6), the user remains the final authority.

### 4. Synthesis stays self-hosted (gemma), accepting a quality cost

Background reflection remains on `gemma4:e2b`; the foreground=OpenAI /
background=local split is unchanged. Cross-linking and reconciliation are the
hardest things for a small CPU model, so the graph will be sparser and rougher
than a frontier model would produce. We accept this to keep background synthesis
free and offline, and lean on tolerant parsing (slugs resolve regardless of
exact phrasing; `_sanitize_events`-style guards) so bad output is cheap. The
reflection prompt is the quality knob.

### 5. No new traversal tool — lean on ambient load

Phase 16 already loads the **entire vault** (bodies, hence `[[links]]`) into the
agent's context each turn, which at this scale strictly dominates link-following
— the model sees every node at once. So Phase 18 adds **no** `related()`/graph-
traversal tool. Explicit traversal earns its keep only when the vault outgrows
full-context load — the same `over_cap` threshold (ADR for Phase 16 / the cap)
that triggers embeddings. The graph's near-term payoff is for the **human**
(navigable `/memory` + Obsidian graph view) and as visible structure in the
loaded context, not as a separate retrieval path.

### 6. The graph *view* is Obsidian's job

We deliberately did **not** build a force-directed graph visualization in the
frontend. "Obsidian is the IDE": the vault is already Obsidian-native, so the
graph view comes for free. The frontend adds only what reading in-app needs —
clickable links and a backlinks section.

## Alternatives considered

| Option | Rejected because |
| ------ | ---------------- |
| Store edges in a link/graph table (or Qdrant) | Dual-store drift on edit/delete; cleanup burden on note delete; reintroduces the two-master problem ADR 007/009 avoid. Links in prose are already the record. |
| Always append (never rewrite) concept pages | Concept pages degrade into incoherent append logs; the whole value is a clean, current page. Reconcile + op-log audit trail is the better trade. |
| Switch synthesis to gpt-4o-mini for better linking | Considered and explicitly declined by the user for Phase 18 to keep background synthesis self-hosted/free; revisit if graph quality proves too weak. |
| Build a graph visualization in the frontend | Duplicates Obsidian's native graph view for a vault that's already Obsidian-compatible; not worth the frontend surface. |
| Add a graph-traversal retrieval tool now | Redundant with Phase 16 full-vault load at current scale; would be machinery serving recall that ambient load already provides. |

---

## Consequences

- **The vault compounds.** Each reflection can create concept/entity pages and
  weave them into the existing graph; `_index.md` tracks the catalog, `_log.md`
  the history.
- **Recall sees structure.** Because Phase 16 loads bodies, the agent reads the
  `[[links]]` as part of standing context — the graph informs reasoning without
  a separate retrieval step.
- **Honest scope boundary.** "Graph only" means the graph grows from
  conversations; documents → wiki synthesis (the dense linking) is the next
  phase's inbox loop. `/lint-wiki` (orphans, broken links, contradictions) is a
  later maintenance phase.
- **Same lineage.** A thin, derived, queryable structure over rich prose — the
  ADR 004 summary-RAG / ADR 009 events lineage, now applied to *relationships*
  between notes.
